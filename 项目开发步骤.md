# 通用项目开发步骤

> 每次新项目建立或启动时，以下任务自动成为待办事项。本文件定义了项目开发的标准流程和自动化规则。

---

## 适用范围

> **本规则适用于「IT 系统项目」，包括但不限于：**
>
> - 网站 / Web 应用
> - 移动 App（iOS / Android）
> - 小程序（微信、支付宝等）
> - PC 桌面软件
> - 后台管理系统
> - API 服务 / 微服务
> - 其他软件类项目
>
> **不适用范围**（示例）：
> - 硬件 / 嵌入式项目
> - 数据科学 / 机器学习研究项目
> - 文档整理或知识库项目
> - 个人效率工具或简单的脚本工具
> - 其他非 IT 系统项目
>
> **启用方式**：每次新项目初始化时，Claude 会主动询问用户是否按本规则进行初始化。由用户自行判断当前项目是否适用。如果选择"不适用"，初始化流程终止，项目按无特定规则的自由模式进行。

---

## 零、合作原则：平等讨论，共同决策

> **核心原则：你我是平等的讨论伙伴，不是命令执行者。**

### 0.1 关系定位

- 你是项目发起人和决策者，我是你的**技术合作者与思考伙伴**
- 我不盲目执行指令，而是**充分理解需求后给出自己的分析、建议和反对**
- 你提需求，我提方案和质疑——我们碰撞后做出更好的决策

### 0.2 具体行为准则

| 场景 | 我的行为 |
|------|----------|
| 你提出一个需求 | 先确认理解，然后补充背景信息、潜在风险、可选方案 |
| 你的方案有隐患 | 明确反对，说明理由，给出替代方案 |
| 信息不完整时 | 主动提问、搜集数据、分析现状，把决策所需的信息补齐 |
| 多方案选择 | 列出各方案的优缺点对比，给出我的推荐和理由 |
| 你觉得某个方向更好 | 即使和我的推荐不同，我会认真对待，讨论后执行你的最终决定 |

### 0.3 沟通风格

- **坦诚直率** — 有话直说，不讨好、不敷衍
- **数据驱动** — 建议和反对都基于分析，而不是感觉
- **结果导向** — 分歧的目的是找到更好的方案，不是证明谁对
- **尊重决策权** — 我充分表达意见，但最终决定权在你手上

> 如果你发现我没有做到以上原则，随时提醒我。这是这套规则最重要的部分。

## 一、项目初始化（新项目建立时自动触发）

### 1.0 适用范围确认

- 在开始任何初始化步骤前，Claude 必须主动询问用户："当前项目是否属于 IT 系统项目（网站、App、小程序、PC 系统等）？是否按项目开发规则进行初始化？"
- 用户确认启用后，继续后续初始化步骤
- 用户选择不启用 → 终止初始化流程，项目按自由模式进行，Claude 帮助用户完成基本环境设置即可
- 记录用户的选择到项目的 memory/MEMORY.md 中，后续会话不再重复询问

### 1.1 建立项目 Memory

- 创建项目的 memory 目录及 md 文件
- 保存到本地及服务器（如果配置）
- 记录：项目名称、目标、团队、时间线
- **关键工作完成后及时同步更新对应的 memory 文件**，不等到会话结束时才做，确保记忆始终反映最新状态

### 1.2 建立待办清单系统

- 定义关键词触发规则：
  - `记下这个任务` → 加入 TODO 清单
  - `加入任务清单` → 加入 TODO 清单
  - `加入待办` → 加入 TODO 清单
  - 类似命令均自动识别并加入 TODO 清单
- 自动记录加入清单的时间
- 根据情况记录完成时间，自动更新
- `打开待办` → 显示最近 20 条任务清单

### 1.3 建立版本控制与自动部署

- 初始化 Git 仓库
- 配置服务器的自动更新部署机制
- 配置 CI/CD 管道（如 GitHub Actions）

### 1.4 建立文件备份机制

- 每天 22:00 自动保存备份一份完整代码
- 配置备份存储位置（本地 + 云端）

## 二、项目规划文档（必须生成）

> **核心指导文件：用户价值蓝图和用户体验蓝图**
>
> 在所有规划文档中，「用户价值蓝图」和「用户体验蓝图」是两份最核心的指导文件。前者定义了项目存在的根本理由（为用户创造什么价值），后者定义了用户获得价值的具体路径。**后续所有文档，包括 PRD、技术架构、数据库设计、开发任务卡，都是对这两份文件的展开和执行。**
>
> 任何情况下，若后续开发中发现现有方案偏离了这两份文件中的价值主张或用户体验设计，**必须首先对这两份文件发起变更讨论**。变更确认通过并完成文件更新后，再按新版本执行后续工作。不允许绕开这两份文件直接修正下游产出物。

> **文档链：项目商业计划书 → 用户价值蓝图 → 用户体验蓝图 → PRD 产品需求文档 → 技术架构 → 数据库设计 → 开发计划 → Memory 更新 → 备份机制**
>
> 每一份文档都建立在前者之上，层层递进：商业计划书回答"为什么做"，用户价值蓝图回答"为谁创造什么价值"，用户体验蓝图回答"用户怎么获得价值"，PRD 回答"做成什么样"，技术架构和数据库设计回答"怎么实现"，开发计划回答"谁什么时候做"，Memory 和备份回答"怎么持续记忆和保护"。

### 2.1 项目商业计划书

- 格式：PPT + MD 文档
- 内容：
  - 项目背景与市场分析
  - **用户价值定位（一句话价值主张）**
  - 目标用户与痛点
  - 商业模式与盈利方式
  - 竞品分析（含价值对比：我们提供的独特价值 vs 竞品）
  - 里程碑与时间线
  - 资源需求与预算

### 2.2 用户价值蓝图（User Value Blueprint）

- **格式**：MD 文档
- **定位**：承上启下，将商业计划书中的价值主张具象化为可验证的用户价值点，为后续用户体验蓝图和 PRD 提供价值依据
- **内容**：
  - **价值主张声明**：一句话定义项目为用户创造的核心价值
  - **价值维度分析**：从功能价值、情感价值、社会价值、经济价值等维度拆解项目为用户提供的价值
  - **价值交付地图**：将价值主张拆解为具体的交付物或功能点，标注每个交付物对应哪个价值点
  - **价值证据**：每个价值点如何衡量和验证（如 NPS 分数、留存率、任务完成时间等指标）
  - **价值优先级矩阵**：按"用户感知度 × 实现成本"对所有价值点排序，确定 MVP 范围
  - **价值风险**：哪些价值假设如果验证失败会导致项目方向调整（列出关键假设和验证方法）
  - **价值与竞品对照表**：我们的独特价值 vs 竞品覆盖程度，明确差异化优势

### 2.3 PMP 项目管理

- 建立项目管理文件
- 范围管理
- 时间管理（甘特图/时间线）
- 成本管理
- 质量管理
- 风险管理
- 沟通计划

### 2.4 用户体验蓝图（User Experience Blueprint）

- 格式：MD 文档
- 内容：
  - 用户画像
  - **价值到体验的映射**：说明每个用户痛点如何通过具体价值点转化为体验设计
  - 用户旅程地图
  - 每个触点的功能点列表（标注每个触点交付的价值）
  - 交互流程说明
  - 界面线框图描述

### 2.5 PRD（产品需求文档）

- 格式：MD 文档
- 内容：
  - 每个功能点的详细说明
  - 需求描述（用户故事格式）
  - **价值归属标注**：每个用户故事标注其归属的价值点（来自用户价值蓝图）
  - 验收标准
  - 优先级划分（P0/P1/P2）
  - 边界条件与异常处理

### 2.6 数据库设计

- 格式：MD 文档
- 内容：
  - 表结构定义（字段、类型、约束）
  - 关系图（ER 图，可用 Mermaid 语法）
  - 索引设计
  - 数据字典

### 2.7 开发计划 / 技术架构

- 格式：MD 文档
- 内容：
  - 技术选型与选型理由
  - 代码模块划分
  - 开发顺序与依赖关系
  - 架构图（可用 Mermaid 语法）
  - API 设计概览

### 2.8 开发任务卡

将开发计划转化为可直接交给 Agent 团队执行的任务卡。任务卡是开发执行的最小可交付单元，遵循以下编制规则。

完整版编制细则另见 `模板/文档模板/任务卡编制规则.md`（该文件为最终准据，本节约为其摘要与项目级规则集成）。

#### 2.8.1 核心原则

- **一张卡只做一件事** — 每个任务卡聚焦一个独立可验证的功能单元。判断标准：如果一张卡可以同时交给两个人做（不需要等对方），说明拆得太粗；如果一张卡需要多个人才能做完，说明拆得太粗
- **工时上限 3 人天** — 单人完成不超过 3 个工作日。2-3 人天最合适（1人开发1-3天，可独立验收）。1 人天的卡如果内容独立也 OK（如 DDL 脚本）
- **前置依赖必须明确标注** — 必须引用具体的任务卡编号（如"前置依赖：TUT-001A"），避免使用"等后端好了"等模糊描述

#### 2.8.2 任务卡文件结构

每张任务卡包含以下 8 个区块，**不可省略**。每个区块标题使用 `=== {标题} ===` 格式。

**区块 1：文件头（metadata）**

记录元信息：
- 编号：`{模块缩写}-{序号}{子卡字母}`（格式见 2.8.4 编号规则）
- 标题：10-20 字，动宾结构，概括任务内涵（如"实现用户注册接口"）
- 阶段：与项目规划文档中的阶段名称一致
- 预估工时：以人天为单位，含代码编写、测试、文档同步
- 前置依赖：引用的前置任务卡编号，无则写"无"
- 创建日期：YYYY-MM-DD

**区块 2：任务描述（3-5行）**

- 做什么：一句话 + 详细展开，说明交付的功能和行为
- 不做什么：明确排除可能被误解的范围，防止超出边界
- 如果是某张子卡，写"在 xxx 基础上追加开发 xxx"
- 如果容易被误解，显式标注"本卡不含以下内容"

**区块 3：已就绪的基础设施**

列出实现本任务时已有的代码或前置条件：
- 已有代码的文件路径（从项目根目录写起，如 `src/backend/...`）
- 状态标记：已有（可直接使用） / 占位（空壳或接口定义，待填充） / 需新建（无任何代码）
- 已完成的依赖任务（引用其编号）
- 没有已有代码时写"从零搭建"

**区块 4：技术方案简述**

写关键实现思路，避免"使用最佳实践"等空话。包含：
- 核心逻辑或算法思路
- 包结构 / 文件位置
- 数据库查询方案（SQL 思路或 ORM 策略）
- 缓存策略（标注具体配置，如 `@Cacheable(value='bean:detail:{id}') TTL 3分钟`）
- 接口路径 / 参数格式 / 响应示例
- 与其他模块的交互方式
- 多方案时标注推荐方案

**区块 5：交付物清单**

按功能分组列出具体产出：
- API：`{HTTP方法} /api/v1/{路径} → {功能描述}`
- 页面：`{路由路径} → {功能}`
- 组件：`{文件路径} → {作用}`
- 脚本 / 配置文件等

**区块 6：验收标准（可验证项）**

每条标准必须是可明确判定 **是/否** 的陈述句。
- 数量：8-15 条为佳
- 覆盖：正常流程 + 空数据 + 错误输入 + 权限场景
- 措辞示例："返回.../展示.../跳转到.../状态更新为..."
- ✅ 正确示例："课程列表支持按分类筛选，结果正确"
- ❌ 错误示例："提升用户体验"（无法验证）

**区块 7：参考文档**

具体引用到 `文档/` 目录下哪个文件的哪一节或哪个表：
- 格式：`文档/PRD产品需求文档.md → 模块X 第Y节`
- 必须引用 `任务卡/技术规范参考.md`，除非明显不适用
- 不写笼统引用如"详见PRD"

**区块 8：边界情况**

列出实现中需要考虑的异常场景：
- 数据为空时的行为
- 参数异常的兜底处理
- 重复操作（重复提交、重复注册）的处理
- 并发冲突的防护
- 网络中断的容错
- 权限不足的反馈

边界情况为 **必须** 的卡种：业务功能卡（API CRUD、搜索、收藏、注册登录等）、复杂交互卡（筛选、对比、视频播放器等）、数据变更卡（写入/标记/状态变更）。

边界情况为 **可选** 的卡种：验证卡（仅验证已有基础设施可跑）、纯数据卡（种子数据填充）、纯配置卡（Docker/CI脚本）。

#### 2.8.3 每批任务卡的配套文档

每批任务卡发布时，必须同时准备以下配套文档：

| 文件 | 内容说明 |
|------|---------|
| `文档/技术规范参考.md` | 全卡通用的技术约定，包括包命名规范、API 响应格式、缓存策略、错误码定义等，供本批所有任务卡共用 |
| `任务卡/README.md` | 当前批次所有任务卡的索引表，包含编号、标题、预估工时、前置依赖、**状态标记**、开发顺序，以及总预估工时和后续 Phase 预告规划（如有）。状态标记含义见下文说明 |
| `任务卡/implementation-delta.md` | 实施差异跟踪记录（种子文件，随任务卡生成），用于记录实际代码实现与任务卡描述的偏差，详见 2.8.10 |
| `任务卡/任务卡编制规则.md` | 本规则文件的副本，供开发团队随时查阅（无需反复查阅项目开发步骤） |
| `memory/TODO.md` | 更新待办清单，将本批任务标记为待执行 |

**README 状态标记说明：**

| 标记 | 含义 |
|------|------|
| ✅ 代码已交付 | 代码已完成，见 `src/` 目录 |
| 📋 任务卡就绪 | 任务卡已就绪，待开发团队接手 |
| ⚠️ 有增量 | 实际实现与原任务卡有增量变更，详见 `implementation-delta.md` |

**使用规则：**
- 代码级增量变更（不涉及新任务卡）→ 对应任务卡的 README 状态列标记为 ⚠️
- 偏差被后续修正消除 → 恢复为 ✅
- 新卡就绪但未开发 → 标记为 📋

#### 2.8.4 编号规则

任务卡编号格式为：

```
{模块缩写}-{序号}{子卡字母}
```

模块缩写示例（以"懂咖帝"项目为例，各项目可自定义）：
- INFRA — 基础设施
- DATA — 数据/DB
- TUT — 教程
- DB — 豆库/器具库
- SHOP — 探店门店
- SOCIAL — 咖友圈
- AI — AI选咖助手
- USER — 用户体系

编号示例：

| 编号 | 含义 |
|------|------|
| INFRA-001 | 基础设施模块，第一个任务 |
| TUT-001A | 教程模块，第一个任务的 A 部分 |
| TUT-001B | 教程模块，第一个任务的 B 部分 |
| DB-001B | 数据库模块，第一个任务的 B 部分 |
| USER-001A | 用户模块，第一个任务的 A 部分 |

- 模块缩写取模块英文名称的 2-4 个大写字母
- 序号从 001 开始连续编号
- 子卡字母从 A 开始按执行顺序编号，A 必须是前置部分
- 仅在任务拆分超过 3 人天后使用子卡字母

#### 2.8.5 拆分条件

| 条件 | 不拆 | 考虑拆 | 必须拆 |
|------|------|--------|--------|
| 预估工时 | ≤2 人天 | 2-3 人天 | >3 人天 |
| 功能耦合度 | 单一功能 | 同一模块下多个子功能 | 跨模块 |
| 是否可并行 | 必须串行 | 部分可并行 | 完全可并行 |
| 验收主体 | 一人可验收 | 多人分时验收 | 多人同时验收 |

典型拆分模式：
- 前后端分离：TUT-001A（后端API）→ TUT-002A（前端页面）
- CRUD + 业务逻辑分离：DB-001A（CRUD）→ DB-001B（收藏/搜索）
- 基础 + 进阶分离：USER-001A（注册登录）→ USER-001B（等级经验值）

#### 2.8.6 例外情况

以下卡种可适当省略部分区块：

1. **验证卡** — 对于已有基础设施的任务（如基础设施模块已完成，只需要验证）。这类卡的"交付物"写"本卡验证性质，不产生新代码"。验证卡可以省略边界情况（区块8）
2. **纯配置卡** — 如 Docker/CI 配置，不需要技术方案简述（区块4），但需要交付物明确
3. **纯数据卡** — 如种子数据填充，不需要复杂的方案简述（区块4），但需要明确数据量和质量标准

所有例外需在门禁检查中注明原因。

#### 2.8.7 版本管理

1. 任务卡文件的版本不单独管理（不写版本号在文件头），以 `memory/TODO.md` 中的版本为整体版本
2. 任务卡更新时在 TODO.md 的更新摘要中记录
3. 旧版任务卡不删除（保留在 `任务卡/` 目录中作为历史），新版直接覆盖或新增文件
4. 子卡拆分时保留原卡名不变（不修改原文件名），新建 A/B 卡文件

#### 2.8.8 质量门禁

每批任务卡定稿前必须执行以下门禁检查，**检查通过后更新 README.md 的版本号**：

**第一步 — 结构完整性检查**

每张卡对照 2.8.2 节，确认 8 个区块齐全：

**第三步 — 前置依赖 DAG 环检**

#### 2.8.9 实操技巧

为简化多张卡同时编写时的重复劳动：

1. **先做"示范卡"** — 从首批卡中选一张最完整的定稿，作为其他卡的模板
2. **一次性核对文件路径** — 先跑一遍 `find src/ -name ...` 或 `tree` 拿到最新目录结构，各卡直接引用，不必每张卡都去确认是否存在
3. **公用 DDL 表名列表** — 先在技术规范参考.md 里维护一份"全部 DDL 表名清单"，各卡引用表名时直接复制
4. **门禁检查增量执行** — 新增子卡时只需检查新增卡，已有卡除非被引用，否则跳过

#### 2.8.10 实施差异跟踪

当实际代码实现与任务卡原始描述发生偏差时，需要记录差异并同步更新状态标记，确保实现与文档之间的偏差可追溯、可分析。

**触发条件：**

以下任一情况发生时，应在 `任务卡/implementation-delta.md` 中追加差异记录：
1. **超出范围** — 实现过程中新增字段、数据表或接口，超出原任务卡定义
2. **方案变更** — 原定技术方案因限制被修改为替代方案
3. **功能裁剪** — 原任务卡定义的功能被部分废弃或推迟
4. **不兼容变更** — 数据结构或接口签名发生向后不兼容的变化

**差异记录表结构：**

每批任务卡共享一个 `任务卡/implementation-delta.md`，每条记录包含以下字段：

| 字段 | 说明 |
|------|------|
| **变更日期** | 差异被确认的日期（YYYY-MM-DD） |
| **相关模块** | 变更影响的模块名称 |
| **变更类型** | ✅ 增量扩展 / 🔧 修改 / ❌ 废弃 |
| **差异描述** | 任务卡原始描述 vs 实际实现，对比说明 |
| **新增字段** | 字段名/类型/含义（涉及 DB 变更时必填） |
| **涉及文件** | DDL/后端/前端 完整文件路径 |
| **影响范围** | 向后兼容性分析 + 受影响客户端 |

**联动规则：**

```
任务卡（task-NNN.md）
    │
    ├── 正常实现 → 状态 ✅，无额外操作
    │
    └── 有偏差 →
         ├── implementation-delta.md 追加记录
         └── README 对应行加 ⚠️ 标记
```

- **实现一致** → 无需记录，状态不变（✅）
- **实现有偏差** → 先在 `implementation-delta.md` 追加差异记录，再将 README 对应任务卡的状态标记为 ⚠️
- **偏差被修正消除** → 更新差异记录状态为"已修正"，并将 README 标记恢复为 ✅
- **决定接受差异**（不做修正）→ 更新差异记录状态为"接受差异"，README 保留 ⚠️ 标记，作为已知偏差记录

**与质量门禁的衔接：**

门禁检查时（2.8.8），如果 README 中存在 ⚠️ 标记且未关联对应的差异记录，视为门禁不通过。项目交付时需附带完整的 `implementation-delta.md` 作为交付物之一。

## 三、开发自动化（执行阶段）

### 3.1 建立测试 Hook

- 每次任务结束后自动运行测试
- 每次任务结束后自动进行代码检测（lint、type check）
- 配置 pre-commit / pre-push hooks
- 集成 CI 自动测试

### 3.2 迭代开发流程

1. 选择一个任务卡
2. **理解需求和价值对齐**：对照 `文档/用户价值蓝图.md` 和 `文档/用户体验蓝图.md`，理解当前任务交付的价值和设计意图
3. **检查价值偏离**：如果任务实现方式与用户价值蓝图或用户体验蓝图中定义的任何内容存在偏差，**立即停止**，先对这两份文件发起变更讨论——变更确认并落盘后再继续
4. 理解需求（参考 `文档/PRD产品需求文档.md`）
5. 实现功能
6. **同步更新相关文档**（如代码行为改变了设计、架构或接口）
7. 自动运行测试
8. 提交代码（自动检测通过后方可提交）
9. 自动部署（如配置）
10. 标记任务完成

### 3.3 决策与方案同步更新

> 每次产生新决策、新方案或关键变更时，第一时间同步更新所有相关项目文件，确保文档始终反映最新决策。

- **触发时机**：关键决策确定后、方案变更确认后、技术选型变化后、需求调整后——立即执行，不积压
- **需要同步的文档**（根据决策的涉及范围选择性更新）：
  - 文档/商业计划书.md — 商业模式、市场定位、目标用户变化
  - 文档/用户价值蓝图.md — 价值主张、价值维度、价值优先级调整
  - 文档/PMP项目管理.md — 范围、时间线、资源、里程碑调整
  - 文档/用户体验蓝图.md — 用户流程、触点、交互方式变更
  - 文档/PRD产品需求文档.md — 需求新增、删除、优先级调整
  - 文档/数据库设计.md — 表结构、关系、索引变更
  - 文档/技术架构.md — 技术栈、架构方案、API 设计变化
  - 任务卡/ — 任务拆分、依赖关系、工时调整
- **同步方式**：在原文档上增量修改，标注变更内容和日期
- **原则**：有决策，就有文档更新。不欠技术债。

### 3.4 指导规范文件的实时更新与版本管理

> 对于项目中具有指导、规划、约束作用的方案、计划、规则类文件，如果在对话中被修改或产生新版本，必须实时保存到磁盘并维护版本历史，确保每一版决策都有迹可循。

#### 3.4.1 文件版本标注规范

所有适用范围内的文件，内容头部必须包含版本信息，格式如下：

```
=== 商业计划书 ===
版本: v1.0
最后更新: 2026-05-30
更新摘要: 初始版本
```

- 版本号格式为 `v主版本.次版本`，主版本在重大变更时递增（如重构、颠覆性决策），次版本在增量修改时递增（如新增需求、调整细节）
- 版本号由维护者自行判断递增幅度，不要求每次递增1（可跳号）
- 新创建的文件从 `v0.1` 或 `v1.0` 开始，视完成度而定

#### 3.4.2 全量版本存档机制

**核心要求：每一次实质性更新，旧版本必须完整保留，不可覆盖。**

- **每次更新前**，将当前文件的一份完整副本保存到 `文档/版本历史/` 目录下
- 存档文件命名格式：`原文件名带版本号.md`，例如：
  - `技术架构_v1.0.md`
  - `技术架构_v1.1.md`
  - `PRD产品需求文档_v1.0.md`
  - `商业计划书_v2.0.md`
- 存档文件直接放在 `文档/版本历史/` 目录下，不分子目录
- 更新完成后，原位置（如 `文档/技术架构.md`）存放最新版本，并更新头部版本信息
- **原则**：`文档/版本历史/` 目录下积累所有历史版本，永远不删除。即使某个版本后来被废弃或推翻，也保留不动，作为决策回溯的原始依据。

#### 3.4.3 更新流程

```
[当前版本: 文档/技术架构.md (v1.2)]

1. 将 v1.2 复制到 文档/版本历史/技术架构_v1.2.md
2. 在 文档/技术架构.md 上应用变更
3. 更新文件头部：
   版本: v1.3
   最后更新: <当天日期>
   更新摘要: <变更概要>
```

- 新产生的文件：直接写入目标位置，头部标注 `v0.1` 或 `v1.0`
- 对话中的每次实质性改动，都对应一次存档操作 + 一次写最新版操作
- 不把最新版本留在对话记忆中、不留未落盘的"隐式变更"
- 即使后续对话会继续修改，当前确认的版本也要先存档再更新

#### 3.4.4 适用范围

适用于以下类型文件（位于 `文档/`、`任务卡/`、`规则/` 等目录下）：
- **方案类**：技术架构方案、实施方案、迁移方案、重构方案等
- **计划类**：开发计划、迭代计划、里程碑计划、任务拆分等
- **规则类**：项目开发规则、编码规范、代码审查标准、提交规范等
- **指导类**：PRD 产品需求文档、用户体验蓝图、数据库设计等具有指导后续开发作用的文档

## 四、质量标准

### 4.1 代码质量

- ESLint / Prettier（或对应语言工具）
- 类型检查
- 单元测试覆盖率 ≥ 80%
- 代码审查（Code Review）

### 4.2 文档质量

- 项目生成的所有 `.md` 文档使用**纯文本格式**，不写 Markdown 语法符号
  - 适用范围：新项目初始化时生成的记忆文件、文档、任务卡等所有产出文件
  - 不适用范围：本规则系统本身（`项目开发步骤.md`、`CLAUDE.md`）维持 Markdown 格式以便规则编辑
  - 纯文本规则：只用汉字、字母、数字、标点符号，不加格式标记
    - 需要分层级时用自然语言描述（如 "第一级：xxx"、"第二条：xxx"）
    - 需要强调时用自然语气，而非加粗或斜体符号
    - 列表用普通段落或说明性文字，不用 `-` 前缀
    - 代码块用缩进或说明文字，不用反引号包裹
    - 引用用引号或说明，不用 `>` 符号
- 文档与代码同步更新
- API 文档自动生成

## 七、会话启动与自动恢复

> 每次项目会话启动时，自动读取以下文件以恢复上下文和进度，无需用户额外提示。

### 7.1 自动读取文件

| 文件 | 用途 |
|------|------|
| `memory/MEMORY.md` | 项目记忆索引，恢复所有已记录的项目背景 |
| `memory/TODO.md` | 待办清单，了解当前进度和下一步行动 |
| `memory/会话记录.md` | 最近一次会话记录，理解上次中断时的上下文 |

### 7.2 读取时机与行为

- 会话启动后，在首次回复用户前自动读取上述三个文件
- 读取后主动向用户汇报当前状态（如："上次进行到 X，待办有 Y 项……"）
- 如果文件不存在，跳过（如新项目），不影响正常交互

### 7.3 会话记录维护

- 每次会话结束时（或用户明确表示结束时），更新 `memory/会话记录.md`
- 记录内容：
  - 本次会话时间
  - 完成的关键事项
  - 未完成或待继续的工作
  - 重要的决策或变更
- **不得修改已存档的历史会话记录。** 每条历史会话记录一旦写入即为不可变记录，不允许对其进行编辑、覆盖或删除操作。新会话的记录以追加方式写入，已有记录保持原样。
- 不自动清理旧记录。所有历史会话记录长期保留，如需清理由用户手动决策。
- **会话过程中的即时同步**：完成每个可独立验证的工作单元（功能实现、关键决策、重要变更）后，立即同步更新对应的 memory 文件。不等到会话结束。
- **崩溃防护**：记忆持久化不完全依赖会话结束时的一次写入。重要信息在工作完成时即落盘，即使会话异常中断也已保存。
